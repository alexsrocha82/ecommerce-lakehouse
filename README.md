# E-Commerce Lakehouse

Production-grade data lakehouse pipeline for an e-commerce domain, running entirely on **Databricks Free Edition** — no Azure credits required.

Demonstrates the complete **Medallion Architecture** (Bronze → Silver → Gold) using industry-standard tools and patterns, from raw file ingestion to analytical star schema.

---

## Stack

| Component | Technology |
|-----------|-----------|
| Compute | Databricks Free Edition (Serverless) |
| Storage | Unity Catalog Volumes (`/Volumes/main/ecommerce_dev/`) |
| File format | Delta Lake (ACID, time travel, MERGE) |
| Ingestion | Auto Loader (`cloudFiles`, directory listing) |
| Language | PySpark + Python 3 |
| Orchestration | Databricks Jobs |
| Catalog | Unity Catalog (Hive Metastore fallback) |

## Job Scheduling

Create two independent Databricks Jobs:

### Job 1 — Daily pipeline
```
Name     : ecommerce-lakehouse-daily
Schedule : 0 0 * * *  (every day 00:00)
Timezone : America/Sao_Paulo

Tasks (each depends on the previous):
  01_bronze_orders       → env=dev, execution_date={{job.start_time.iso_date}}
  02_bronze_customers    → env=dev, execution_date={{job.start_time.iso_date}}
  03_silver_orders       → env=dev, execution_date={{job.start_time.iso_date}}
  04_silver_customers    → env=dev, execution_date={{job.start_time.iso_date}}
  05_gold_all            → env=dev, mode=incremental
  06_quality_analytics   → env=dev
```

### Job 2 — Weekly maintenance
```
Name     : ecommerce-lakehouse-optimize
Schedule : 0 2 * * 0  (Sunday 02:00)
Timezone : America/Sao_Paulo

Tasks:
  07_optimize → env=dev, tables=all, dry_run=false, vacuum_hours=168
```

> Failure in Job 2 never blocks Job 1 — they are fully independent.

## Key Design Decisions

### PII Tokenization (not masking)

All sensitive fields (email, CPF, phone, customer_id) are replaced with deterministic SHA-256 tokens:

```python
SHA256(f"{SALT}:{value}").hexdigest()[:16]
```

**Why tokenization instead of masking:**
- Masking destroys data irreversibly (`u***@gmail.com` — no recovery possible)
- Tokenization is reversible by anyone who holds the salt
- Enables audit trails, fraud investigation, and compliance workflows

**Why tokenize customer_id (not personally identifiable by itself):**
- `CUST-0001` alone reveals nothing
- But exposing it in gold allows analysts to join with external CRM systems and reconstruct full PII outside the pipeline's control
- `customer_token` is a **joinability firewall** — prevents accidental re-identification

> In production: store the salt in Azure Key Vault and access via `dbutils.secrets.get()`.

### SCD Type 2+6

- **Type 2**: new row inserted for every attribute change — full history preserved
- **Type 6**: `current_segment` updated across ALL rows (historical + current)
- Two columns answer different questions:
  - `segment` → "what was the segment during this specific period?"
  - `current_segment` → "what is the segment today, regardless of the period?"

### No Surrogate Keys

Gold dimensions use natural keys (`customer_token`, `product_id`) directly — no integer surrogate keys (SK).

**Reasoning:** Kimball-style integer SKs were designed for legacy columnar databases (Teradata, Netezza, 1990s-2000s) where integer joins were significantly faster than string joins. In modern Delta Lake, ZORDER + predicate pushdown + bloom filters make string joins equally performant. `monotonically_increasing_id()` in PySpark is also non-sequential and unreliable as a stable SK.

### OPTIMIZE runs weekly, not daily

`OPTIMIZE + ZORDER` rewrites **all files** in the table — not just the new ones. Running it on every small incremental append means paying the cost of rewriting millions of rows to optimize a few hundred new ones. The right pattern: append daily, compact weekly.

## Differences vs Production Environment

| Component | Production | This project (Free Edition) |
|-----------|-----------|---------------------------|
| Storage | ADLS Gen2 | UC Volumes (`/Volumes/main/...`) |
| Auto Loader mode | file notification (Event Grid) | directory listing |
